3. OBJECT-ORIENTED DESIGN
===============================================

3.1 Architecture Overview

The system employs a layered architecture with clear separation of concerns:

PRESENTATION LAYER:
- LibraryConsoleApp: Console-based menu-driven interface
- LibraryManagementGUI: JavaFX GUI with tabbed interface

BUSINESS LOGIC LAYER:
- LibraryService: Core business rules (availability checking, overdue calculations)
- ValidationUtils: Input validation and data integrity enforcement

DATA ACCESS LAYER:
- DatabaseManager: SQLite operations and persistence

DOMAIN MODEL LAYER:
- Book: Represents library books with encapsulated properties
- Member: Represents library members (students/staff)
- BorrowRecord: Represents borrowing transactions

3.2 Object-Oriented Principles Application

ENCAPSULATION:
Each entity class (Book, Member, BorrowRecord) encapsulates its data with private fields and public getter/setter methods, enabling controlled access and validation. The DatabaseManager encapsulates all database complexity, exposing only necessary operations.

Example implementation:
```java
public class Book {
    private int bookId;
    private String title;
    private String author;
    private String category;
    private String availabilityStatus;
    
    // Encapsulated access through getters/setters
    public String getTitle() { return title; }
    public void setTitle(String title) { this.title = title; }
}
```

ABSTRACTION:
The DatabaseManager abstracts database implementation details. Consumers interact through high-level methods (addBook, updateMember, getOverdueRecords) without concerning themselves with SQL queries or connection management.

```java
public boolean addBook(Book book) {
    String sql = "INSERT INTO books ...";
    // Implementation details hidden
}
```

SINGLE RESPONSIBILITY:
- ValidationUtils handles input validation exclusively
- LibraryService manages business calculations
- DatabaseManager manages data persistence
- UI classes handle presentation logic only

3.3 Design Patterns Employed

MODEL-VIEW-CONTROLLER (MVC):
While not a pure MVC, the architecture separates:
- Model: Book, Member, BorrowRecord classes
- View: LibraryConsoleApp, LibraryManagementGUI
- Controller: DatabaseManager, LibraryService

REPOSITORY PATTERN:
DatabaseManager acts as a repository, providing an abstraction layer between business logic and data access.

DAO (DATA ACCESS OBJECT) PATTERN:
Database operations are centralized in DatabaseManager, providing a consistent interface for all persistence operations.

3.4 Class Relationships

The system employs composition and association:
- BorrowRecord associates with Book (via bookId) and Member (via memberId)
- DatabaseManager manages collections of all entity types
- UI layers depend on DatabaseManager for data operations

4. IMPLEMENTATION DETAILS
===============================================

4.1 Database Design

The SQLite database utilizes three normalized tables:

BOOKS TABLE:
- book_id (PRIMARY KEY): Unique identifier
- title (TEXT NOT NULL): Book title
- author (TEXT NOT NULL): Author name
- category (TEXT NOT NULL): Subject category
- availability_status (TEXT NOT NULL): Available/Borrowed

MEMBERS TABLE:
- member_id (PRIMARY KEY): Unique identifier
- member_name (TEXT NOT NULL): Member full name
- email (TEXT NOT NULL): Contact email
- membership_type (TEXT NOT NULL): Student/Staff

BORROW_RECORDS TABLE:
- record_id (PRIMARY KEY): Unique transaction identifier
- book_id (INTEGER NOT NULL): Foreign key reference
- member_id (INTEGER NOT NULL): Foreign key reference
- borrow_date (DATE NOT NULL): Transaction date
- due_date (DATE NOT NULL): Expected return date
- return_status (TEXT NOT NULL): Borrowed/Returned/Overdue

Data integrity is enforced through:
- Primary keys ensuring uniqueness
- NOT NULL constraints for required fields
- Data type validation
- Foreign key relationships

4.2 CRUD Implementation

CREATE OPERATIONS:
addBook(), addMember(), addBorrowRecord() use PreparedStatements to prevent SQL injection and handle parameterized queries safely.

```java
public boolean addBook(Book book) {
    String sql = "INSERT INTO books (title, author, category, availability_status) VALUES (?, ?, ?, ?)";
    try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
        pstmt.setString(1, book.getTitle());
        // ... additional parameters
        pstmt.executeUpdate();
        return true;
    } catch (SQLException e) {
        return false;
    }
}
```

RETRIEVE OPERATIONS:
Multiple retrieval methods support different access patterns:
- getAllBooks(): Complete list retrieval
- getBookById(): Direct access by identifier
- searchBooksByTitle(), searchBooksByAuthor(): Search functionality
- filterBooksByCategory(): Category-based filtering
- getOverdueRecords(): Complex query for business logic

UPDATE OPERATIONS:
updateBook(), updateMember(), updateBorrowRecord() methods reconstruct objects with modified properties and persist changes:

```java
public boolean updateBook(Book book) {
    String sql = "UPDATE books SET title = ?, author = ?, category = ?, availability_status = ? WHERE book_id = ?";
    try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
        pstmt.setString(1, book.getTitle());
        // ... additional parameters
        pstmt.setInt(5, book.getBookId());
        pstmt.executeUpdate();
        return true;
    }
}
```

DELETE OPERATIONS:
deleteBook(), deleteMember(), deleteBorrowRecord() remove records from the database with confirmation prompts in the UI layer to prevent accidental deletion.

PRESENTATION LAYER:

6.1 System Strengths

ROBUST ERROR HANDLING:
Comprehensive exception handling prevents crashes and provides user feedback.

SCALABILITY:
Database-driven architecture supports growth in books, members, and records.

MAINTAINABILITY:
Clear separation of concerns makes code modifications straightforward.

USER-FRIENDLY:
Both console and GUI interfaces accommodate different user preferences and skill levels.

DATA INTEGRITY:
Validation at multiple levels prevents invalid data from persisting.

FLEXIBILITY:
Service layer enables future business rule modifications without UI changes.

6.2 Limitations and Future Improvements

PERSISTENCE OPTIMIZATION:
Current implementation loads entire datasets into memory. For libraries with millions of books, pagination or lazy loading would be beneficial.

CONCURRENCY:
While the architecture supports multi-threading, the current implementation lacks thread synchronization for concurrent database access. Production deployments should implement connection pooling and thread-safe operations.

AUTHENTICATION:
No user authentication system exists. Production version should require login with role-based access control.

REPORTING LIMITATIONS:
Current reports are text-based. Future versions could include:
- Statistical charts and graphs
- PDF export functionality
- Scheduled report generation
- Email distribution of reports

NOTIFICATION SYSTEM:
Missing automated notifications for:
- Overdue book reminders
- Due date approaching alerts
- System events
